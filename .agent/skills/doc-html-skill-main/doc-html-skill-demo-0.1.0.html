rf-mcp

Where AI agents meet Robot Framework

OwnerRobotMCPAudienceTest & automation engineers · QA professionals · The Robot Framework community Schemav2

Summary

  • ✓rf-mcp is the handshake between your AI agent and Robot Framework. Describe a test in plain words, get a working .robot suite back.
  • ✓It does not guess. Keywords run in a live Robot Framework context, step by step, with real feedback before a single line is written down.
  • ✓Three execution backends — in-process, attach-to-live, and full CLI runs — because one size never fits all.
  • ✓Built on FastMCP and Robot Framework 7, with small-LLM optimizations so lighter models keep up.
  • ✓Web, mobile, API, database. If it needs automating, it is probably in scope.

#Key decisions

Run keywords in a live, in-process RF context

adopted

Keeps execution stateful and interactive — live variables, real page source, immediate feedback — instead of writing code blind and hoping.

ADR-020

Offer three execution backends, not one

adopted

In-process for speed and state, attach-to-live for reusing a debugger session, CLI for full and dry suite runs.

Discover keywords by reflection, not runtime Libdoc

adopted

Read live library instances directly with Python introspection — faster, and it sees exactly what is loaded.

Optimize for small LLMs

adopted

Tool profiles, intent-based actions, and response/state compression keep token budgets honest.

#Constraints

  • high MCP STDIO shares one JSON-RPC channel, so Robot Framework console output is redirected and suppressed to keep the stream clean.
  • medium Requires Robot Framework 7.0+ — it leans on the RF 7.x running APIs.
  • medium The attach bridge listens on 127.0.0.1:7317 and expects an X-MCP-Token header.
  • info Browser, mobile, and database features need the matching install extra.

#Interfaces

  • MCP over STDIO — command "uv run -m robotmcp.server"
  • MCP over HTTP — http://localhost:8000/mcp
  • Attach bridge — HTTP on 127.0.0.1:7317
  • Console entry point — robotmcp = robotmcp.server:main

#Deployment context

Runs as a local MCP server right next to your editor, or in Docker (ghcr.io/manykarim/rf-mcp), exposing MCP on port 8000 and an optional Django dashboard for watching sessions in real time.

#Alternatives considered

Generate .robot files straight from a prompt

Generate .robot files straight from a prompt — rejected. No validation, brittle output, and it breaks the moment a locator moves.

Subprocess-only execution

Subprocess-only execution — rejected. You lose live state and the step-by-step feedback that makes this useful.

#The short version

Robot Framework is the Swiss Army knife of automation. rf-mcp hands that knife to your AI agent without anyone getting hurt. It is a Model Context Protocol server that turns a plain-language test idea into a real, reviewable Robot Framework suite — by discovering keywords, executing them interactively against a live execution context, checking the results, and only then writing the final .robot file. Web, mobile, API, database: same workflow, different libraries.

#Why this matters

Generated tests are validated against a live RF context before they are written down — no blind code generation, no brittle output.

#What you get

Natural-language test generation. Interactive, stepwise keyword execution with real-time DOM and page-source inspection. Multi-library support for Browser, Selenium, Appium, Requests, and Database libraries. BDD and data-driven generation when you want Given/When/Then or parameterized templates. Optional persistent memory that recalls successful steps, locators, and known fixes across sessions. A debug attach bridge to reuse a live execution context from your IDE. And an optional dashboard, because sometimes you just want to watch it work.
